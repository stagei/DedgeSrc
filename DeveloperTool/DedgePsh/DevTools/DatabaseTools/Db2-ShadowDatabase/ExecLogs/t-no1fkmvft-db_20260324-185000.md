### PrimaryKey Name Mismatch — Explained

**What the report says:** 1866 primary key constraints are "Missing in Target".

**What this actually means:** The primary keys **exist** on the same tables with the same columns — but they have **different system-generated names**.

DB2 auto-names unnamed primary key constraints using the pattern `SQL` + a timestamp. The name encodes when the constraint was created. Example:

| Source (FKMVFT, created 2026-03-22) | Target (FKMVFTSH, created 2026-03-24) |
|---|---|
| `DBM.SQL260322174232390` | `DBM.SQL260324205351xxx` |
| `DBM.SQL260322174232740` | `DBM.SQL260324205351yyy` |

The `260322` = March 22 (when the source DB was last restored from PRD).
The `260324` = March 24 (when Step 2 ran `CREATE TABLE ... ADD PRIMARY KEY` on the shadow DB).

The verification script compares constraint names as strings. Since the names differ, it reports 1866 "missing" — but every table has its primary key. The columns and enforcement are identical. Only the auto-generated name changed.

**Why this happens:** The shadow pipeline recreates tables using DDL extracted from the source. The DDL says `ADD PRIMARY KEY ("COL1", "COL2") ENFORCED` without naming the constraint. DB2 assigns a new `SQL<timestamp>` name at creation time.

**Impact:** None. No data loss, no structural difference, no missing constraints. This is inherent to how DB2 names unnamed constraints and cannot be avoided without explicitly naming every PK in the source DDL.
