## Errors and Fixes

### Error 1: Empty view DDL text in replay list (PID 1828)

**Root cause:** All capture functions (`Get-IndexDdlFromCatalog`, `Get-GrantStatementsFromCatalog`, `Get-DependentViewDdlFromCatalog`, FK functions) used `return ,$statements` which wraps the array with the unary comma operator. When `$statements = @()` (empty array), `return ,@()` sends a single pipeline object (the empty array itself). The caller's `@()` wrapping creates a 1-element array containing the empty array, which resolves to an empty string when iterated.

**Fix applied:** Added `Where-Object` filter to remove null/empty entries after capture:
```powershell
$indexStatements = @($indexStatements | Where-Object { $null -ne $_ -and ("$_").Trim() -ne '' })
```

**Result:** Filter prevented phantom entries, but the deeper nested-array issue remained (see Error 2).

### Error 2: DB21034E invalid SQL statement (PID 4248)

**Root cause:** Even with the empty-entry filter, `return ,$statements` for non-empty arrays (e.g., 2 grant statements) created a nested array: `@(@("GRANT A", "GRANT B"))`. Iterating produced one element (the inner array), which when string-interpolated became `"GRANT A GRANT B"` (space-joined) — invalid SQL.

**Fix applied:** Removed the `,` (unary comma) from all 7 capture function `return` statements:
```powershell
# Before: return ,$statements
# After:  return $statements
```

Without the comma, PowerShell pipeline-emits individual strings, and `@()` at the call site correctly collects them as a flat array.

**Result:** Grants replayed correctly as individual `db2 "GRANT ..."` commands.
