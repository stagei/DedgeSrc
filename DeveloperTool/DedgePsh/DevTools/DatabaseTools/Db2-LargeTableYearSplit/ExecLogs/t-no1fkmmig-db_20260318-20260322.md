## All Errors Discovered

### 1. SQL1013N — Database Not Found (Run 3, March 18 10:10)

```
SQL1013N  Databasealiaset eller databasenavnet "FKMMIG" ble ikke funnet.
```

**Cause:** User was restoring the database while the job was still running.  
**Impact:** Job failed (`JOB_FAILED`). Non-recurring.  
**Fix:** N/A — user-initiated restore.

### 2. CMD.exe `>=` Mangling (Runs 8-9, March 18 21:54 - 22:47)

```
db2 export ... select * from DBM.D365_BUNTER where YEAR(T_TPKT_VISMA)  1>2025
```

**Cause:** The `>=` operator in the `WHERE` clause was interpreted by `CMD.exe` as a file redirect (`1>2025` creates a file named "2025"). The IXF output file was 0 bytes.  
**Impact:** EXPORT produced empty files. Two runs killed.  
**Fix:** Changed `WHERE YEAR(col) >= $CutoffYear` to `WHERE YEAR(col) BETWEEN $CutoffYear AND 9999`.

### 3. SQL0750N — Dependent View Blocks RENAME (Run 10, March 18 23:26)

```
DB21034E  Kommandoen ble behandlet som en SQL-setning, fordi den var ugyldig
SQL0750N  The table cannot be renamed because it is referenced by dependent views.
```

**Cause:** `RENAME TABLE` fails when dependent views exist. View `TV.RPA_AVSTEMMING_BUNTER` references `DBM.D365_BUNTER`.  
**Impact:** RENAME failed, original table unchanged. Export IXF files cleaned up correctly.  
**Fix:** Added Step 4b/7 to DROP all dependent views before RENAME, then recreate them in Step 7.

### 4. "Cannot index into a null array" (Run 11, March 18 23:55)

```
RenameAndReload failed for DBM.D365_BUNTER: Cannot index into a null array.
```

**Cause:** In the `DROP VIEW` loop, `$viewDdl` was a nested array (not a plain string). The `-match` operator didn't populate `$matches` when applied to an array, so `$matches[1]` threw a null-index error.  
**Impact:** DROP VIEW failed, RENAME never attempted. No data loss.  
**Fix:** Added defensive `($viewDdl | Out-String).Trim()` conversion before regex matching in all view DDL loops (drop, recreate, and error recovery).

### 5. SQL0104N — Multi-Statement Index/Grant Recreation (Runs 12-13, March 19 + 22)

```
SQL0104N  Symbolet "CREATE INDEX DBM.D365_BUNTER_I1 ON DBM.D365" etter ...
SQL0104N  Symbolet "GRANT SELECT ON DBM.D365_BUNTER TO BYODDR" etter ...
SQL0104N  Symbolet "ALTER TABLE DBM.VISMA_TRANSER ADD PRIMARY K" etter ...
SQL0104N  Symbolet "GRANT CONTROL ON DBM.VISMA_TRANSER TO DB2NT" etter ...
```

**Cause:** All captured DDL statements (24 indexes, 12 grants) are concatenated into a **single** `db2 "..."` command. Db2 CLP only accepts one SQL statement per invocation. The first statement might succeed, but the parser fails on the second statement's token.  
**Impact: CRITICAL** — The script reports "Recreated 1 index(es)" and "Recreated 1 grant(s)" but in reality:
- **VISMA_TRANSER** (Run 13): 8 indexes captured → only 1 `db2` invocation attempted → at most 1 index created, 7 missing. Same for 5 grants.
- **D365_BUNTER** (Run 12): 24 indexes captured → only 1 `db2` invocation → at most 1 index created, 23 missing. Same for 12 grants.
- Dependent views were recreated correctly (only 1 view, so single-statement worked).

**Status:** **NOT YET FIXED.** Tables are currently **missing indexes and grants** after RenameAndReload.

## Improvements Required

### P1: CRITICAL — Fix Multi-Statement Index/Grant Recreation

**Current bug:** `Get-IndexDdlFromCatalog` and `Get-GrantStatementsFromCatalog` return arrays of DDL strings, but `Move-AllYearsRenameAndReload` concatenates all statements into a single `db2 "..."` call. Db2 CLP rejects multi-statement input.

**Fix:** Execute each DDL statement individually via separate `Invoke-Db2MultiStatement` calls:

```powershell
foreach ($idxDdl in $indexStatements) {
    $cmd = @("db2 `\"$idxDdl`\"")
    try {
        $null = Invoke-Db2MultiStatement -WorkObject $WorkObject -Db2Commands $cmd -IgnoreErrors
        $idxRecreated++
    }
    catch {
        Write-LogMessage "  Index recreation failed: $($_.Exception.Message)" -Level WARN
    }
}
```

Same pattern for grants. This must be done before the next production run.

**Impact of not fixing:** Tables will be missing all but possibly the first index and grant, causing severe query performance degradation and broken application access.

### P2: HIGH — Validate Object Counts After Recreation

Add post-recreation validation: query `SYSCAT.INDEXES` and `SYSCAT.TABAUTH` on the new table and compare counts against the captured DDL arrays. Log a WARNING if counts don't match.

### P3: MEDIUM — Add Index Build Progress Logging

For tables with many indexes (D365_BUNTER has 24), log each index name as it's created so progress can be monitored and failures identified individually.

### P4: LOW — Run RUNSTATS After LOAD

After `db2 load` with `nonrecoverable`, table statistics are stale. Add `RUNSTATS ON TABLE <schema>.<table> WITH DISTRIBUTION AND DETAILED INDEXES ALL` to ensure the optimizer uses accurate statistics.

### P5: LOW — Cleanup Year Tables After Transfer to HST

Year tables (VISMA_TRANSER_1, _2008, _2010, _2019, _2024, D365_BUNTER_2022) from ExportLoad runs remain in the database. These should be transferred to BASISHST or dropped after verification.
