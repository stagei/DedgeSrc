## Errors and Fixes

### Error 1: DB21034E wrapping SQL0601N on index creation (run 1)

**Root cause:** `Invoke-Db2MultiStatement` threw on DB21034E, which wrapped SQL0601N ("name already exists"). The PK index `ALTER TABLE ADD PRIMARY KEY` generated a system index, and subsequently trying `CREATE INDEX` with the same name from the captured DDL collided.

**Initial fix:** Added try/catch for DB21034E, treating it as "already exists" and counting as `$idxSkip`. This was **incorrect** — the real root cause was discovered later (see Error 4).

### Error 2: SQL0104N duplicate CREATE OR REPLACE VIEW prefix (run 2, D365_BUNTER)

**Root cause:** `Get-DependentViewDdlFromCatalog` reads `SYSCAT.VIEWS.TEXT` which can start with comment lines (e.g. `-- ViewName source`). The `^CREATE` regex failed to match because of the leading comments, so the function prepended another `CREATE OR REPLACE VIEW`, resulting in `CREATE OR REPLACE VIEW ... CREATE OR REPLACE VIEW ...` — invalid SQL.

**Fix:** Strip leading comment lines (`^--.*`) from the view DDL text before the `^CREATE` regex check.

### Error 3: SQL0911N lock timeout during index DROP from _TMP (remediation)

**Root cause:** An orphan db2 process from a previous failed run was holding locks on D365_BUNTER_TMP.

**Fix:** `db2stop force` + `db2start` to clear all connections. DROPs succeeded immediately after restart.

### Error 4: All 30 user-defined indexes missing — the real root cause

**Root cause:** `RENAME TABLE X TO X_TMP` moves the table AND all its named indexes. Index names (`DBM.TABLE_I1`, etc.) are globally unique within a schema. After RENAME, the new table (created via `CREATE TABLE LIKE`) has no indexes. When Step 7 tries `CREATE INDEX DBM.TABLE_I1`, Db2 returns SQL0601N because that name is still on the _TMP table. The try/catch from Fix 1 silently swallowed these as "already exists" — but they existed on the WRONG table.

**Fix (permanent):** Before creating indexes in Step 7, DROP each named index from the _TMP table first. This frees the name for the new table. The catch block was removed; index creation now fails hard on any error.

**Additional improvement:** Captured DDL is now persisted to a JSON file (`*_captured_ddl.json`) immediately after capture, before any destructive operations. A generic `Replay-CapturedDdl.ps1` script can read this file for recovery instead of hardcoded ad-hoc scripts.
